# GoForKids — iPad app

Native iPad build of GoForKids. Wraps the existing React frontend in a
WKWebView, replaces Render-hosted KataGo inference with on-device CoreML
running on the Neural Engine.

**Status (Phase 2A, 2026-05-01):** working end-to-end on real M-series
iPads. AI moves + score lead computed locally. UI still loads from Render;
game state still lives on Render. See **Roadmap** below.

## Why this directory looks weird

The Xcode project is built on top of the
[ChinChangYang/KataGo](https://github.com/ChinChangYang/KataGo) fork — a
250 MB tree containing the C++ engine + iOS Xcode project that we vendor
in `ios/KataGo/`. That subdirectory is **gitignored** (too big and we
don't change the upstream sources).

What IS committed:
- `ios/swift/` — the source of truth for our Swift bridge code
  (`KataGoBridge.swift`, `ContentView.swift`) + tweaked `default_gtp.cfg`
- `ios/AppIcon-1024.png` — app icon (placeholder, regenerated by
  `generate_icon.py`)
- `ios/generate_icon.py` — icon generator script
- This README

To build, you re-acquire the vendored fork and copy our Swift files into it
(see Setup below).

## Setup (first-time clone)

### Prerequisites
- Xcode 16+ installed
- Apple Developer account (free personal team works for sideloading)
- An M-series iPad to deploy to (Simulator can't run CoreML — see gotcha #6
  in the spike playbook)

### 1. Clone the fork

```sh
cd ios/
git clone --depth 1 --branch v1.16.4-coreml1 https://github.com/ChinChangYang/KataGo.git
```

This pulls ~111 MB. The fork bundles a working iOS Xcode project at
`ios/KataGo/ios/KataGo iOS/KataGo iOS.xcodeproj`.

### 2. Apply our Swift overrides

```sh
cp swift/KataGoBridge.swift "KataGo/ios/KataGo iOS/KataGo iOS/"
cp swift/ContentView.swift  "KataGo/ios/KataGo iOS/KataGo iOS/"
cp swift/default_gtp.cfg    "KataGo/ios/KataGo iOS/Resources/"
```

### 3. Download the model files

```sh
cd "KataGo/ios/KataGo iOS/Resources"
gh release download v1.16.4-coreml1 --repo ChinChangYang/KataGo \
  --pattern "kata1-b28c512nbt-adam-s11165M-d5387M-null.bin.gz" \
  --pattern "KataGoModel19x19fp16-adam-s11165M.mlpackage.zip"
mv kata1-b28c512nbt-adam-s11165M-d5387M-null.bin.gz default_model.bin.gz
unzip -q KataGoModel19x19fp16-adam-s11165M.mlpackage.zip
mv KataGoModel19x19fp16-adam-s11165M.mlpackage KataGoModel19x19fp16.mlpackage
rm KataGoModel19x19fp16-adam-s11165M.mlpackage.zip
```

### 4. Open the project + apply build gotchas

```sh
open "KataGo/ios/KataGo iOS/KataGo iOS.xcodeproj"
```

The fork's project comes pre-configured with most build settings, BUT
several gotchas have to be re-applied for a fresh clone. The full playbook
lives at `~/Projects/GoForKidsIOS-Spike/README.md` (the original spike
documentation). The 8 gotchas in summary:

1. **`COMPILE_MAX_BOARD_LEN=29` → set to `19`** (Build Settings → Preprocessor
   Macros, both `KataGo iOS` and `katago` targets, both Debug + Release)
2. **Add two missing C++ files to the `katago` target's Compile Sources:**
   `cpp/neuralnet/sgfmetadata.cpp` and `cpp/search/evalcache.cpp`
3. **Use the non-m1 mlpackage** (we downloaded the right one in step 3)
4. **Rename mlpackage to `KataGoModel19x19fp16.mlpackage`** (Swift loader
   builds the filename dynamically without an `m1` suffix — done in step 3)
5. **Ship the raw `.mlpackage`, not the auto-compiled `.mlmodelc`:**
   - Remove `KataGoModel19x19fp16.mlpackage` from Copy Bundle Resources
   - Add a Run Script build phase (above Copy Bundle Resources, "Based on
     dependency analysis" UNCHECKED) that copies the raw mlpackage:
     ```sh
     SRC="${SRCROOT}/KataGo iOS/Resources/KataGoModel19x19fp16.mlpackage"
     DST="${TARGET_BUILD_DIR}/${UNLOCALIZED_RESOURCES_FOLDER_PATH}/KataGoModel19x19fp16.mlpackage"
     rm -rf "${DST}"
     mkdir -p "$(dirname "${DST}")"
     cp -R "${SRC}" "${DST}"
     ```
6. **Build Settings → "User Script Sandboxing" → No** (otherwise the Run
   Script can't write into the .app bundle)
7. **iOS Simulator can't actually run inference** — build verification only;
   real `genmove` only works on a device
8. **Add `KataGoBridge.swift` to the `KataGo iOS` target** (after copying
   from `ios/swift/`, drag into the Project navigator and confirm Target
   Membership)

Bundle Identifier defaults to `ccy.KataGo-iOS` (the fork's). Change to
`com.patrickdowell.goforkids` if/when you want a phasesix-branded ID
(re-triggers signing setup).

### 5. Add the icon

`Assets.xcassets` → `AppIcon` → drag `ios/AppIcon-1024.png` into the
1024 slot.

### 6. Add the "Bundle React frontend" Run Script (Phase 3)

The iPad app loads its UI from a bundled copy of the React frontend
(NOT from Render). A Run Script build phase rebuilds the frontend and
copies `dist/` into the app bundle on every Xcode build.

1. Project navigator → `KataGo iOS` (blue icon) → target `KataGo iOS` → **Build Phases**.
2. Top of Build Phases area, click **`+`** → **New Run Script Phase**.
3. **Drag it above "Copy Bundle Resources"** (and below the existing "Copy raw mlpackage" Run Script — order doesn't actually matter between the two, but keeps things tidy).
4. Rename the phase to **"Bundle React frontend"**.
5. Paste this script:
   ```sh
   set -e

   FRONTEND_DIR="/Users/patrickdowell/Projects/GoForKids/frontend"
   DST="${TARGET_BUILD_DIR}/${UNLOCALIZED_RESOURCES_FOLDER_PATH}/web"

   echo "=== Bundle React frontend phase ==="
   echo "  frontend: ${FRONTEND_DIR}"
   echo "  dst:      ${DST}"

   if [ ! -d "${FRONTEND_DIR}" ]; then
     echo "ERROR: frontend directory missing at ${FRONTEND_DIR}"
     exit 1
   fi

   # Build with Render API as the backend (HTTP fetches still go to Render
   # for game state; the bridge handles AI inference locally).
   cd "${FRONTEND_DIR}"
   export VITE_API_BASE_URL="https://goforkids-api.onrender.com"
   # Xcode build env doesn't include Homebrew/NVM by default. Add common
   # node install locations so npm resolves.
   export PATH="/opt/homebrew/bin:/usr/local/bin:${HOME}/.nvm/versions/node/$(ls ${HOME}/.nvm/versions/node 2>/dev/null | tail -1)/bin:${PATH}"
   npm run build

   # Copy dist/ into the app bundle's web/ subdirectory.
   rm -rf "${DST}"
   mkdir -p "${DST}"
   cp -R dist/* "${DST}/"

   echo "Bundle complete:"
   ls "${DST}"
   ```
6. **Critical checkboxes:**
   - ⬜ **"Based on dependency analysis"** → **UNCHECKED** (so it runs every build — frontend changes don't trigger Xcode dependency tracking)
   - ⬜ **"Show environment variables in build log"** → CHECKED (helps debug PATH issues)

If `npm run build` fails with "command not found", check the build log to see what `${PATH}` resolved to and add your node install path.

### 7. Build & run on iPad

`⌘R` to your connected iPad. First launch incurs a one-time CoreML model
compilation (~30s on M1; cached afterward in Application Support).

## Architecture (Phase 3)

```
┌──────────────────────┐                ┌──────────────────────┐
│      iPad app        │                │       Render         │
│                      │                │                      │
│  WKWebView (file://) │                │  goforkids-api       │
│        │             │                │  FastAPI + KataGo    │
│        ▼             │  game state    │  (CPU, b20)          │
│  Bundled React app   │  /move /pass   │                      │
│  (frontend/dist      │ ◄────────────► │                      │
│   shipped in app)    │                └──────────────────────┘
│        │             │
│        ▼             │
│  window.kataGo       │
│        │             │  AI inference is fully on-device.
│        ▼             │  Render is only used for game-state
│  KataGoBridge.swift  │  endpoints (board, captures, ko).
│        │             │
│        ▼             │
│  KataGoHelper.mm     │
│  (ObjC++ → C++)      │
│        │             │
│        ▼             │
│  CoreML on ANE       │
└──────────────────────┘
```

Single React codebase (`frontend/src/`) serves both web (Render-deployed)
and iPad (bundled). `frontend/src/api/client.ts` detects `window.kataGo`
and routes AI inference through the bridge instead of HTTP. AI rank
selection is done by `frontend/src/ai/moveSelector.ts` reading
`data/profiles/b28.yaml` — same source the Python backend uses.

UI ships with the iPad app, so the React assets work even if Render is
down. The backend (game state) still requires Render to be reachable —
making the iPad fully offline is Phase D.
